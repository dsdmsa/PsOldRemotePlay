## open-rp Source Code Analysis

**Registration is completely skipped** — requires importing `.orp` file from a real PSP.

### .ORP File Format
- Magic: "ORP", version 2
- Contains: skey0, skey1, skey2 (system keys), pkey (16-byte shared secret), PSP MAC, PSP OpenPSID, PS3 MAC, PS3 nickname
- The pkey is generated BY THE PS3 during registration — it's random, not derived

### Session Key Derivation (confirmed matching our implementation)
```
xor_pkey  = pkey XOR skey0       → AES key
xor_nonce = nonce XOR skey2      → AES IV
auth      = AES-CBC-Encrypt(PSP_MAC padded to 16, key=xor_pkey, iv=xor_nonce)
PREMO-Auth = Base64(auth)
```

### Key Insight
The PSP does NOT implement the registration crypto — it uses the PS3's built-in registration via physical connection (USB for PSP, WiFi AP for Phone/PC). Only the SESSION crypto is in open-rp. The registration protocol is entirely PS3-side code.

## PS3 AES-CBC Behavior

- Standard OpenSSL-style: **IV buffer IS modified in-place** after encrypt/decrypt
- In the registration handler, the SAME `&uStack_80` buffer is used for both decrypt (request) and encrypt (response)
- This means the response is encrypted with the IV **after decryption** (last ciphertext block), NOT the original IV
- The client must compute the same post-decryption IV to decrypt the response
- For testing: first try decrypting with original IV; if that fails, the client may need to track the IV state

- Text segment base: found by searching for known key bytes (VERIFIED: 0x01B60000)

## NEW: PS4 Registration Protocol Comparison (Critical Clue!)

From [kingcreek/ps4-remote-play](https://github.com/kingcreek/ps4-remote-play) — the PS4 registration crypto is SOLVED and shows structural parallels:

### PS4 Key from PIN:
```python
expanded_pin = struct.pack('!L', pin) + b'\x00' * 12  # 4-byte BE PIN + 12 zero bytes
key = [x ^ y for (x, y) in zip(REG_AES_KEY, expanded_pin)]
```

### PS4 Nonce Derivative:
```python
deriv[i] = NONCE_KEY[i] ^ ((0x200 + nonce[i] - i - 0x29) & 0xFF)
```

### Structural Parallels with PS3:

| Aspect | PS3 (PREMO) | PS4 (RP) |
|--------|-------------|----------|
| Endpoint | `/sce/premo/regist` | `/sce/rp/regist` |
| Port | 9293 | 9295 |
| PIN → key | XOR with static key + subtract offset | XOR with static key |
| Body prefix | 480 bytes for PC/VITA type | 480 bytes padding |
| Magic offset | Phone: 0x28, PSP: 0x25, PC: 0x2B | 0x29 |
| Key response | `PREMO-Key` | `RP-Key` |
| Encryption | AES-128-CBC | AES-128-CFB |

### Key Insight
The PS4 uses `struct.pack('!L', pin)` — the PIN as a **4-byte big-endian unsigned int** — NOT as an 8-byte longlong. If the PS3 follows a similar pattern, the IV context might be:
- `PIN as 4-byte BE uint + 4 zero bytes` (not 8-byte longlong)
- OR the PIN might be XOR'd with a static key before use
- The 480-byte body prefix for PC/VITA type matches EXACTLY between PS3 and PS4

This PS4 code is the closest public reference to what the PS3 registration crypto likely looks like.

**Source:** https://github.com/kingcreek/ps4-remote-play/blob/master/ps4rp/crypto.py
